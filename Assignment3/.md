if (x > 0) {
    if (x < 100) {      // this is the problematic branch
        do_something();
    }
}
```

Now think about some **later branch** further down in the code, trying to use history to predict itself. It looks back at recent history and sees three different situations:

**Case 1 — x > 0 AND x < 100 (both branches execute):**
```
A → outer(taken) → inner(taken) → B → C → [target]
```

**Case 2 — x > 0 AND x >= 100 (inner branch skipped):**
```
A → outer(taken) → B → C → [target]
```

**Case 3 — x <= 0 (outer branch not taken, inner never reached):**
```
A → outer(nottaken) → B → C → [target]
```

The inner if **sometimes exists in the history and sometimes doesn't.** When it appears, it pushes B and C one slot further back. When it disappears, B and C return to their original positions.

---

### Why This Destroys TAGE

TAGE works by hashing the history register to index into prediction tables. That hash depends on **exact bit positions.**

When the inner branch executed:
```
position:   1  2  3  4  5
bits:       1  1  1  0  1
                      ^
                      B is at position 4
```

When the inner branch was skipped:
```
position:   1  2  3  4
bits:       1  1  0  1
                   ^
                   B is at position 3
```

TAGE computes a **completely different hash** for what is functionally the same situation. It ends up storing the same pattern in two different table entries, training each one half as often, and never converging on a confident prediction. The more branches like this in the code, the worse the problem gets — each one multiplies the number of "phantom" history patterns TAGE has to deal with.

---

### The Three Properties That Make a Branch Problematic

A branch causes misalignment when it has all three of these:

1. It appears **conditionally** in the execution path — not always executed
2. Other branches come **after it** within the history window — so its presence shifts their positions
3. It has **no direct correlation** with the branch you are trying to predict — it is just noise sitting between useful signal and the target

Compilers generate these constantly without intending to cause problems:

- Null checks: `if (ptr != null)` wrapping a block
- Bounds checks: `if (i < array.length)` inserted automatically
- Type checks: `if (obj instanceof Foo)` in object-oriented code
- Short circuit evaluations: `if (a && b && c)` generates multiple branches where later ones are skipped if earlier ones fail
```
